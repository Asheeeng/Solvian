# Solvian（Copilot）项目需求分析

## 1. 项目概述
Solvian 是一个基于 Java 17 + Spring Boot 3.2.5 构建的 AI 数学逻辑诊断系统，面向数学题图像识别、过程分析、错误定位和教学反馈场景。系统当前以“业务链路完整、可演示、易扩展”为目标，已经具备从认证登录到异步诊断、结果展示、反馈回收、历史统计和报告下载的完整业务闭环。

项目采用静态前端页面 + Spring Boot 后端接口的结构，前端通过统一的 `storage.js` 封装调用后端 API，后端通过控制器、服务、异步任务处理器、MyBatis-Plus Mapper 与 PostgreSQL 数据库协同工作。系统还支持接入智谱 GLM 视觉模型与文本推理模型，并提供模型连通性测试接口。

## 2. 建设目标
本项目的核心建设目标如下：

1. 构建一个可运行的数学题 AI 诊断演示系统。
2. 支持多角色登录，并体现角色权限差异。
3. 支持教师上传题目图片并发起 AI 异步诊断。
4. 支持前端查看诊断进度、诊断结果、错误定位与反馈信息。
5. 支持对 AI 识别结果进行人工反馈，形成闭环优化数据。
6. 支持历史记录、错题本、统计看板与报告下载能力。
7. 通过数据库持久化保存认证、任务、诊断结果、反馈和事件日志。
8. 为后续扩展真实业务字段、权限体系和模型能力预留架构空间。

## 3. 用户角色分析

### 3.1 学生
学生角色主要面向“查看与反馈”场景，核心诉求是：
- 注册并登录系统；
- 查看自己产生或有权限访问的诊断记录；
- 查看历史结果与错误说明；
- 对诊断结果提交“准确/不准确”反馈；
- 下载自己有权访问的报告。

### 3.2 老师
老师是当前系统的核心业务使用者，核心诉求是：
- 登录系统进入工作台；
- 上传题目图片并查看预览；
- 触发 AI 诊断流程；
- 查看异步诊断进度与最终结果；
- 查看错误步骤、错误标签与推理说明；
- 提交 AI 反馈，标记识别是否准确；
- 查看错题本与统计面板；
- 下载报告；
- 发起模型连通性测试。

### 3.3 管理员
管理员角色当前主要用于体现更高权限与全局数据访问范围，核心诉求是：
- 登录系统；
- 在更高权限范围内查看记录与统计信息；
- 作为全局访问者参与权限边界管理。

## 4. 功能需求分析

### 4.1 认证与账户管理
系统需支持：
- 用户注册；
- 用户登录；
- 用户退出登录；
- 根据登录 Token 获取当前用户信息；
- 基于角色识别用户身份；
- 前端保存登录态并在接口调用时携带认证头。

约束：
- 注册和登录需校验用户名、密码、角色；
- 未登录或会话失效时需返回未授权提示；
- 非登录/注册 API 统一走鉴权拦截。

### 4.2 题目上传与工作台交互
系统需支持：
- 在首页工作台上传数学题图片；
- 显示题目原图与局部高亮预览；
- 支持启发式（Socratic）模式开关；
- 对上传、预览等动作进行事件上报。

约束：
- 仅教师角色可真正发起诊断；
- 学生或其他角色不可使用老师端诊断主能力。

### 4.3 AI 异步诊断
系统需支持：
- 前端提交题目图片创建诊断任务；
- 后端为任务分配 `task_id`；
- 对图片做预处理与输入存储；
- 将任务写入 `diagnosis_task` 表；
- 将任务放入异步执行器进行后台处理；
- 先进行视觉识别，再进行推理分析；
- 在任务执行过程中逐步更新任务状态、进度和阶段信息；
- 前端轮询任务接口并实时展示结果；
- 诊断完成后生成 `diagnosis_record` 记录。

### 4.4 诊断结果展示
系统需支持：
- 展示诊断状态；
- 展示步骤重构结果；
- 展示反馈说明与错误原因；
- 展示错误索引与标签；
- 展示图像高亮区域；
- 区分处理中、完成、失败等状态。

### 4.5 AI 反馈回收
系统需支持：
- 用户对某次诊断结果提交“准确 / 不准确”反馈；
- 当选择“不准确”时，必须提供错误类型；
- 可附加备注说明；
- 回写 `diagnosis_record` 当前记录中的反馈字段；
- 额外写入 `ai_feedback_record` 用于统计与审计。

### 4.6 历史记录与统计分析
系统需支持：
- 查询历史诊断记录；
- 以错题本样式展示近期记录；
- 统计错误类型分布；
- 统计 AI 反馈分布；
- 展示最近诊断记录。

权限要求：
- 学生只能查看自己的记录；
- 管理员与更高权限用户可查看更广范围数据。

### 4.7 报告下载
系统需支持：
- 基于 `recordId` 下载 PDF 报告；
- 报告下载需进行权限控制；
- 当前实现允许返回占位 PDF 内容，但接口链路必须完整。

### 4.8 事件埋点与运行支撑
系统需支持：
- 记录上传、预览、启动诊断、完成诊断、查看错题本、查看统计、提交反馈等行为事件；
- 保存事件类型、页面、动作、关联记录和载荷信息；
- 提供模型连通性测试能力，用于验证模型可用性。

## 5. 非功能需求分析

### 5.1 可用性
- 页面需要具备完整的上传、提示、按钮状态、错误提示和结果可视化反馈。
- 前端需支持浅色/深色主题切换并保持体验统一。

### 5.2 可扩展性
- 后端分层明确，控制器、服务、任务处理器、模型客户端、数据访问相互解耦。
- 支持未来扩展更多题型、更多权限规则和真实 PDF 报告能力。

### 5.3 可维护性
- API 能力按控制器划分明确；
- 数据模型职责清晰；
- 异步任务与最终结果分表存储，便于维护和追踪。

### 5.4 安全性
- 通过 `AuthInterceptor` 统一鉴权；
- 敏感模型密钥要求从环境变量或配置注入，不允许硬编码；
- 不同角色的数据访问能力需受限制。

### 5.5 性能与响应
- 诊断主链路采用异步任务，避免长时间阻塞同步请求；
- 前端通过轮询获取进度，增强用户可感知性；
- 支持视觉识别结果缓存复用以降低重复计算成本。

## 6. 核心业务流程分析

### 6.1 主流程：老师异步诊断
1. 老师登录系统；
2. 进入首页工作台并上传题目图片；
3. 前端调用 `POST /api/diagnose` 创建任务；
4. 后端完成图片预处理、输入存储和任务落库；
5. 队列执行器异步触发任务处理器；
6. 视觉 Worker 先进行图像识别；
7. 推理 Worker 再进行错误分析与结果整理；
8. 结果写入 `diagnosis_record` 并更新 `diagnosis_task`；
9. 前端轮询 `GET /api/diagnose/{taskId}` 获取状态；
10. 页面渲染最终步骤、反馈、标签与错误定位；
11. 老师可继续提交反馈、查看历史/统计并下载报告。

### 6.2 辅助流程：学生查看与反馈
1. 学生登录系统；
2. 查看自己可访问的历史记录；
3. 打开对应诊断结果；
4. 提交 AI 反馈；
5. 下载自己有权访问的报告。

### 6.3 支撑流程：模型测试
1. 老师登录；
2. 调用模型测试接口；
3. 后端访问模型服务并返回连通性测试结果。

## 7. 技术实现特点分析

### 7.1 前端特点
- 使用静态 HTML + CSS + ES Module；
- `login.html`、`register.html`、`home.html` 分别承担认证与工作台场景；
- `storage.js` 统一处理请求、登录态、Token 注入、错误处理；
- `home.js` 负责诊断轮询、反馈提交、错题本/统计展示、报告下载等主工作流。

### 7.2 后端特点
- 基于 Spring Boot 提供 REST API；
- `AuthController`、`DiagnosisTaskController`、`HistoryController` 等控制器各司其职；
- `AuthServiceImpl` 管理用户和会话；
- `DiagnosisTaskServiceImpl` 管理任务创建与任务状态；
- `DiagnosisTaskProcessor`、`VisionWorker`、`ReasoningWorker` 组成异步诊断流水线；
- `ModelClientServiceImpl` 编排视觉与推理模型调用；
- `HistoryServiceImpl`、`DashboardServiceImpl`、`AiFeedbackServiceImpl`、`EventLogServiceImpl` 提供记录聚合与审计支持。

### 7.3 数据特点
数据库覆盖以下核心实体：
- 用户表 `app_user`
- 会话表 `app_session`
- 诊断记录表 `diagnosis_record`
- 诊断任务表 `diagnosis_task`
- AI 反馈表 `ai_feedback_record`
- 事件日志表 `event_record`

其中：
- `diagnosis_task` 用于描述任务过程；
- `diagnosis_record` 用于描述最终结果；
- `ai_feedback_record` 与 `event_record` 用于优化和审计；
- 多个字段使用 TEXT 存储结构化 JSON 内容，满足轻量持久化与后续迁移要求。

## 8. 接口需求概览
系统当前核心接口如下：

- `POST /api/auth/register`：注册
- `POST /api/auth/login`：登录
- `POST /api/auth/logout`：退出登录
- `GET /api/auth/me`：获取当前用户
- `POST /api/diagnose`：创建异步诊断任务
- `GET /api/diagnose/{taskId}`：查询诊断任务状态
- `POST /api/ai-feedback`：提交 AI 反馈
- `GET /api/history`：查询历史记录
- `GET /api/dashboard-summary`：查询统计面板
- `GET /api/report/{recordId}/pdf`：下载报告
- `POST /api/log-event`：记录事件日志
- `GET /api/model/test`：模型连通性测试

## 9. 结论
从当前代码与数据库设计来看，Solvian 已经形成了“认证 -> 诊断 -> 结果 -> 反馈 -> 统计 -> 报告”的完整业务闭环。项目虽然仍保留占位实现（如 PDF 报告内容），但从系统分析、架构表达、数据库建模和业务时序来看，已经具备完整项目的分析价值。
